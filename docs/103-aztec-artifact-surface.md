# Aztec Artifact Surface

This document explains how the historical Aztec artifact lane fits into the
current Symbolic Web framing introduced in
[Symbolic Web Definition](./01-symbolic-web-definition.md).

The short form is:

```text
canonical symbolic state
-> realized artifact surface
-> Aztec projective embodiment
-> lattice / mesh composition
-> scan / re-entry
```

The purpose of this note is to place the older Aztec work correctly inside the
current handbook, so it is read as a materialized artifact lane under the same
canonical law rather than as a separate competing theory.

## 0. Core Statement

In the artifact lane, the Aztec code is the shared self-defining artifact
surface: a flat 2D projective embodiment of runtime state that can be directly
embedded, rasterized, scanned, and instantiated across environments.

This means the Aztec surface is not merely a decorative rendering. It is a
portable materialized witness surface.

At the same time, it does not replace canonical symbolic law. It sits below the
canonical event and transcript layer, and above environment-specific file,
render, and scan embodiments.

## 0.5 Realization Axis

The deterministic witness lane and the scanner-compatible lane should be read
as two positions on the same realization axis, not as competing artifact
systems.

The clean rule is:

```text
one canonical artifact
-> witness realization
-> scanner realization
```

linked by:

```text
offset transform
```

The realization axis is the coordinate space in which one invariant artifact
can appear in multiple lawful surfaces.

An offset transform is a deterministic mapping between the canonical witness
lattice and a scanner-compatible Aztec realization such that:

- payload bytes remain identical
- transcript remains identical
- symbol order remains identical
- continuation and block boundaries remain identical
- artifact identity and path remain identical

Only the following may differ by transform:

- coordinate placement
- framing and wrapper conventions
- scanner-visible quiet zone or barcode constraints
- scan and re-entry method

## 1. Relationship To The Symbolic Web

The [Symbolic Web Definition](./01-symbolic-web-definition.md) freezes the main
rule of the project and explicitly allows portable artifact surfaces as one of
the ways canonical symbolic state leaves one environment and re-enters another:

```text
signal
-> symbol
-> transcript
-> projection
-> interpretation
```

The Aztec artifact surface fits into that stack like this:

```text
canonical event and transcript
-> materialized artifact surface
-> environment-specific embodiment
-> scan / re-entry
```

So the Aztec lane should be read as:

```text
one artifact and witness layer inside the Symbolic Web
```

not:

```text
a separate protocol authority
```

The practical consequence is:

- `01` defines the apex law
- this note defines one materialized artifact embodiment of that law
- scan and decode are re-entry operations back into the same symbolic system

## 2. Surface Versus File

One important distinction must stay explicit:

```text
Aztec surface != arbitrary file container
```

The correct split is:

- Aztec surface = the shared 2D projective artifact
- raster, SVG, print, or embedded file = one embodiment of that surface
- scan = lawful re-entry from embodiment back into runtime form

So the project should not say:

```text
the file is the artifact
```

It should say:

```text
the Aztec surface is the artifact
the file is one transport embodiment of that surface
```

## 3. Authority Boundary

The historical artifact lane only fits the current project if the authority
boundary stays clear.

The current handbook already says:

- Braille is canonical emitted carrier
- transcript is the public proof surface
- projections may not redefine transcript

The Aztec lane therefore must be read as:

```text
canonical symbolic state first
Aztec surface second
```

That means:

- the Aztec artifact is authoritative as a shared projective embodiment for
  exchange and instantiation
- but it does not outrank canonical symbolic replay or transcript law

## 4. Historical Layering

The historical implementation already implied a clean layering:

```text
AtomicKernel
-> Artifact
-> lattice placement
-> Aztec sheet or symbol surface
-> scan and decode
```

The important point is that placement and embodiment were already treated as
derived from prior encoding law, not as free-form UI invention.

That historical layering fits the current project very well:

```text
Braille / transcript law
-> canonical payload or event state
-> artifact surface
-> lattice placement
-> Aztec embodiment
-> scan / re-entry
```

The historically implemented lossless path was more specific:

```text
artifact bytes
-> A13 frame encoding
-> 60-slot Aztec symbols
-> canonical 27x27 surface table
-> decode
-> exact byte recovery
```

That exact deterministic path is what makes the artifact lane more than a
generic export format.

The standardized scanner realization may add a transport wrapper or
scanner-specific envelope, but only under an offset agreement that preserves
the same canonical payload and transcript.

## 5. Aztec As Projective Kernel

The Aztec artifact can be read as a projective kernel because it preserves
canonical identity while allowing multiple environment-specific embodiments.

This means:

- the same underlying state can be rasterized, embedded, printed, tiled, or
  projected
- scan can re-instantiate the same state in another environment
- the artifact is stable under transport as a 2D projective surface

That is why the older language around a “kaleidoscope filter” can be normalized
more carefully as:

```text
projective symmetry kernel
```

The artifact preserves identity while supporting transport and composition
through a stable surface law.

## 6. Lattice And Mesh Composition

The artifact lane also matters because individual Aztec surfaces can be
composed.

The clean rule is:

```text
Aztec artifacts are composable projective kernels that can be arranged in
lattices, stacks, or meshes while preserving independent decode and canonical
identity.
```

This means:

- one artifact can be decoded on its own
- multiple artifacts can be tiled or meshed together
- composition must not destroy independent decode

That is what makes them usable as transport surfaces across multiple
environments instead of only as one-off images.

## 7. Scan And Re-Entry

Scanning is not an incidental convenience. It is the lawful act of
instantiation.

The re-entry path is:

```text
shared Aztec surface
-> scan
-> decode
-> recover canonical state
-> continue runtime or projection
```

This is why the Aztec surface belongs inside the Symbolic Web: it allows a
materialized artifact to leave one environment and re-enter another without
requiring meaning-level agreement.

Under the offset agreement model the equality rule is:

```text
decode(witness realization) == decode(scanner realization)
```

## 8. Why This Matters

The Aztec artifact surface matters because it gives the project a real portable
materialization lane.

It allows:

- direct embedding in another environment
- printing and scanning
- raster and SVG transport
- later extrusion or higher-dimensional projection

while keeping the artifact tied back to canonical symbolic state.

This is stronger than a generic “image export” story. It is a lawful artifact
story.

## 9. Template Artifact Mode

Template and builder surfaces may expose the Aztec lane as an artifact mode for
selected nodes, entities, or schema objects.

The intended browser-side cycle is:

```text
active authoring object
-> freeze as artifact
-> Aztec SVG or PNG embodiment
-> export
-> import or scan
-> deterministic re-instantiation
```

This means:

- authoring state remains a projection/editor surface
- artifact export must be derived from canonical payload
- SVG import must re-enter through canonical payload recovery
- PNG import may re-enter through the deterministic Aztec PNG header/metadata lane
- scanner-compatible barcode import may re-enter through the standardized
  offset-linked transport realization
- the template UI does not become protocol authority

## 10. Current Reading In The Handbook

The safest current reading is:

```text
canonical symbolic event and transcript
-> artifact surface
-> Aztec embodiment
-> scan / decode
-> re-instantiated event or payload
```

This keeps the historical artifact lane alive without allowing it to replace
the newer canonical event law.

## 11. One-Sentence Freeze

The system defines one canonical artifact class whose deterministic witness
surface and scanner-compatible transport surface are offset-linked realizations
of the same payload, transcript, and symbol order.
