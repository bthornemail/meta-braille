# Matroid Emergence

This document explains why matroids appear in the project as a necessary law of
the system rather than as an imported mathematical decoration.

The short form is:

```text
signal
-> points
-> closure
-> basis
= matroid
```

## 0. Core Statement

Matroids emerge in this project because the runtime needs a deterministic law
for:

- independence
- redundancy
- closure
- minimal representation

Once the system is built from finite canonical points and requires lawful ways
to decide when a set is minimal, dependent, or closed, a matroid-like layer is
forced.

## 1. Historical Path

The historical implementation path was not:

```text
choose matroids first
-> force them onto the system
```

It was:

```text
carrier and artifact system
-> automaton point extraction
-> closure and independence requirements
-> basis and circuits
-> matroid layer
```

That path is explicit in the historical Haskell layering:

```text
AtomicKernel
-> Artifact
-> Automaton
-> Matroid
-> configuration and projection routines
```

## 2. Automaton Produces The Ground Set

The automaton layer already defines a canonical point-bearing world.

In the historical implementation:

- the automaton accepts canonical carriers
- applies incidence timing
- resolves state
- returns a canonical point or a carrier reducible to one

This means the system already has:

```text
finite point-set candidates
```

Those points are the natural ground set for later closure and independence
reasoning.

So the first forced step is:

```text
stream or carrier
-> canonical points
```

## 3. Deterministic Construction Forces Closure

Once points exist, the system still needs to decide:

- when a point is generated by other points
- when a set is independent
- when a basis is minimal
- when a circuit is the smallest dependent set

The historical matroid layer makes that explicit:

- closure maps
- independence tests
- basis selection
- circuits
- rank and nullity
- flats and closed sets

The important project-specific detail is that this layer is:

```text
finite
constructive
canonically ordered
```

That is exactly what a deterministic symbolic runtime needs.

## 4. Why This Was Necessary

The project did not need matroids for abstraction alone.

It needed them because multiple carriers, placements, or projections could still
refer to the same resolved state.

So the runtime needed a lawful answer to questions like:

- is this representation redundant
- is this point already generated by others
- what is the smallest faithful support
- which sets are dependent

That is precisely where closure and basis enter.

So the correct reading is:

```text
matroid = minimal law of symbolic dependency
```

## 5. Fano And Finite Relational Closure

The earlier geometry work gave the system a finite relational kernel.

That matters because:

- the system is not open-ended in an uncontrolled way
- circuits can be finite and inspectable
- stepping and incidence remain bounded

So the historical Fano-style work and the matroid layer are compatible:

```text
Fano-like finite relational closure
-> point and incidence structure
-> matroid-style closure and basis law
```

This does not mean the current project is “just a Fano matroid.” It means the
finite relational kernel made a constructive closure law possible and useful.

## 6. Artifact And Lattice Layers Confirm The Need

The artifact and lattice layers strengthened the need for a matroid law.

Those layers already introduced:

- multiple geometric carriers
- normalized artifact identities
- canonical placement tables
- lattice and mesh composition
- independently decodable projection surfaces

Once the same signal can appear in multiple lawful embeddings, the runtime must
still answer:

- which embeddings are equivalent
- which support set is minimal
- which placements are redundant
- how composition preserves identity

Those are matroid-shaped questions.

So the later geometric and artifact work did not weaken the matroid layer. It
made it more necessary.

## 7. Current Reading In The Symbolic Web

The current project can now read the matroid layer as part of the Symbolic Web
stack:

```text
Braille and transcript
-> canonical event and point-bearing runtime
-> closure / independence / basis law
-> geometry, artifact, and projection surfaces
```

This means matroids belong in the structural layer, not in the projection layer.

They are a law for:

- dependence
- equivalence
- closure
- minimal support

not a law for:

- visual arrangement
- observer interpretation
- semantic narration

## 8. Relationship To The Axis Model

The axis model remains:

```text
7-axis   = stepping
240-axis = identity and topology
256-axis = observer projection
```

The matroid layer sits inside the structural logic that binds those axes
without becoming a fourth rival axis.

It helps determine:

- which point-sets are minimally sufficient
- which structures are closed
- which carriers are dependent or redundant

So the safe reading is:

```text
matroid law = structural dependency layer across canonical symbolic points
```

## 9. Practical Reduction

The most practical reduction for the current handbook is:

```text
signal
-> canonical points
-> closure
-> basis
-> artifact and projection
```

That is the reason matroids appeared.

The project did not discover them accidentally. It encountered the minimal law
required for deterministic symbolic dependency.

## 10. One-Sentence Freeze

Matroids arise in this project as the minimal deterministic law governing
closure, independence, redundancy, and basis over canonical symbolic point-sets
derived from carrier-bearing streams.
